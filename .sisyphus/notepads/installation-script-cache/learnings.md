# Learnings

- `InstallScriptService` now owns the hardcoded install-script repository constants (`sesori-ai/sesori_apps_monorepo`), script paths (`install.sh`, `install.ps1`), and cache TTL (`5 * 60 * 1000`) instead of relying on config.
- Release selection is intentionally strict: only non-draft, non-prerelease GitHub releases whose `tag_name` starts with `bridge-v` and whose `published_at` parses into a valid date are eligible.
- Task 2 keeps both script bodies in one atomic cache entry shaped as `{ tag, installSh, installPs1, expiresAt }`, so `getInstallSh()` and `getInstallPs1()` always read from the same release snapshot.
- Warm refresh failures intentionally preserve the previous cache contents and push `expiresAt` forward by one TTL window, which throttles retries while continuing to serve stale install scripts.
- Task 3 replaces the refresh seam with direct unauthenticated GitHub API calls: release discovery walks `/releases?per_page=100&page=n` sequentially until the `Link` header no longer advertises `rel="next"`, then the winning `bridge-v...` tag is reused for both contents requests.
- Cache refresh now rechecks releases before touching script bodies; if the selected tag matches the cached tag, the service skips both `/contents/{path}?ref={tag}` fetches and only extends the TTL, preserving Task 2's single shared cache entry behavior.
- Task 4 wires `InstallScriptService` through the app composition root and shared test harness only; no route registration is needed yet, so the service is carried as a clean DI dependency for the next task.
- The new wiring test can stay focused on boot-time DI by stubbing the MongoDB connector/accessor layer at the harness boundary, which avoids requiring a live local MongoDB instance just to prove the override and default paths compile and boot.
- Task 5 follows the existing `/auth/public-key` precedent for plain-text responses: the install route handlers stay thin, call `InstallScriptService` directly, and use `reply.type("text/plain").send(scriptBody)` without JSON wrappers or auth hooks.
- Task 6 proves the install endpoints end-to-end by mounting the real `InstallScriptService` in route tests, mocking `fetch` plus `Date.now`, and asserting both `/install.sh` and `/install.ps1` move together across a cache refresh.
- Route-level cold-cache failures still surface through Fastify’s shared error handling as `502` JSON `{ "error": "bad_gateway" }`, so the route test only needs to assert the transport contract, not the internal `BadGatewayError` shape.
- When a test only overrides a subset of service methods, `Pick<InstallScriptService, "getInstallSh" | "getInstallPs1">` keeps the stub honest while a narrow cast at the DI boundary satisfies the helper signature.
- Hands-on QA on this branch passed for all install-focused suites: `tests/install/install-script-service.test.ts`, `tests/install/routes.test.ts`, and `tests/install/wiring.test.ts` all exited cleanly and covered the happy path, atomic tag refresh pairing, warm-cache stale-on-error, cold-cache `502`, and DI boot paths.
- The route suite is already the repo’s direct execution pattern for these endpoints: it boots Fastify through `createTestApp(...)`, exercises the live HTTP handlers with `app.inject(...)`, and verifies the real `InstallScriptService` behavior without requiring external GitHub or MongoDB access.
